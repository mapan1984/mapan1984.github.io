应用程序启动参数

* `-s <module>`：启动指定 module 并调用其 `start/0` 函数。
* `-s <module> <function>`：启动指定 module 并调用 `function/0` 函数。
* `-s <module> <function> <arg1> <arg2> ...`：启动指定 module 并调用 `function/1` 函数，参数以列表 `["arg1", "arg2", ...]` 的形式传入。
* `-eval <expr>`：直接在启动时执行一段 Erlang 表达式。

代码加载和路径参数

* `-pa <dir>`：将目录添加到代码加载路径列表，可以通过 `code:get_path()` 查看
* `-pz <dir>`：将目录添加到代码加载路径列表的开头，可以实现优先加载
* `-boot <file>`：指定启动是使用的 boot 文件，用于启动 OTP 应用
* `-config <file>`：指定启动时使用的配置文件

节点相关参数

* `-name <name>`：设置节点的长名称，格式 `nodename@hostname`，在分布式 Erlang 中，nodename 唯一，hostname 可解析。
* `-sname <name>`：设置节点的短名称，同一局域网内的节点通信。
* `-setcookie <cookie>`：设置分布式节点之间的共享 cookie，节点必须使用相同的 cookie 才能互相通信。

运行模式

* `-noshell`：禁用 Erlang shell，适用于以非交互模式运行脚本
* `-noinput`：禁止从标准输入读取数据
* `-detached`：以后台模式运行
* `-verbose`：启用详细输出
* `-remsh <node>`：连接到远程节点的 shell

其他

* `--`：用于分隔 Erlang VM 参数和普通参数，例如 `erl -sname mynode -- arg1 arg2` 中 arg1, arg2 是普通参数，可以通过 `init:get_plain_arguments/0` 获取

### 变量/原子

变量特点：

- 不可变(Immutable)
- 动态类型：声明时不需要指定类型
- 强类型：不会自动做类型转换


匹配/绑定 `=`：

- `=` 在 Erlang 中表示匹配（Erlang 没有赋值）
- 如果匹配时左侧变量没有值，将右侧的值绑定到左侧的变量上
- 如果左侧、右侧无法匹配，抛出异常报错


命名规则：

- 变量首字母必须大写
- 首字母小写被视为原子 `atom` 类型
- `_` 特殊保留字，可以匹配任何值（通常用于在模式匹配中丢弃某些值）
- `_` 前缀开头的变量，忽略编译器的未使用警告


变量示例：

    1> One = 1.
    1
    2> Two = One + One.
    2
    3> Number = One.
    1
    4> Number = Two.
    ** exception error: no match of right hand side value 2
    5> _ = One.
    1
    6> _ = Two.
    2
    7> _Ignore = One.
    1
    8> _Ignore = Two.
    ** exception error: no match of right hand side value 2


原子 `atom` 类型的作用类似于其他语言中枚举类型，一般只用于比较是否相同，具体是什么值无关紧要

    1> atom.

    3> atom = 'atom'.

## Module

- module 名必须与文件名一致

创建文件 `useless.erl`

``` erlang
%% 声明 module name
-module(useless).

%% export 是导出语法，指定导出 add/1, add/2, add/3 函数
%% 没导出的函数在 Module 外是无法访问的。
-export([add/1, add/2, add/3]).

% add/1
% 通过 erl -noshell -run 命令执行，参数为列表
% 执行 halt() 区别 Erlang VM 执行完退出
add([A, B]) ->
  io:format("~p~n", [A ++ B]),
  erlang:halt(0).

% add/2
% 函数可以根据参数类型重载
add(A, B) when is_number(A), is_number(B) ->
  A + B;
add(A, B) when is_list(A), is_list(B) ->
  A ++ B.

% add/3
add(A, B, C) ->
  A + B + C.
```

编译

    mkdir ebin
    erlc -o ebin useless.erl

    2> useless:add("acb", "def").

## 递归

Erlang 中没有 while loop，因此需要用递归实现循环操作。

递归计算列表长度

``` erlang
len([]) -> 0;    %% 空列表的长度是 0
len([_ | T]) -> 1 + len(T).  %% 列表的长度是 1 + 剩余列表的长度
```

尾递归计算列表长度

``` erlang
len(L) -> len(L, 0).  %% 这其实只是给 len/2 的第二个参数设置了一个默认值 0.

len([], Acc) -> Acc;  %% 所有的元素都读完了
len([_ | T], Acc) -> len(T, Acc+1).  %% 读一个元素，Acc 增 1，然后计算剩下的 List 的长度
```

## 进程

Erlang 进程不是指操作系统的进程，而是由 Erlang 虚拟机自身调度管理的进程（类似于其他语言的协程）

每个进程有自己的信箱，进程间可以通过发送消息通信。

- 使用 `erlang:spawn/1,2,3,4` 创建 Erlang 进程，返回 PID（进程 ID）
- 使用 `<PID> ! <data>` 向进程 PID 发送消息，操作符左侧为 PID，右侧为任何 Erlang 数据
- 使用 `receive ... end` 从自己的信箱中读取消息
- 使用 `self/0` 返回当前进程的 PID

Erlang shell 本身也是作为普通进程实现的。

    % erl
    1> self().
    <0.84.0>
    2> self() ! hello.
    hello
    3> flush().
    Shell got hello
    ok

常用的进程相关机制

- `register/2` 可以将进程注册一个名称，之后可以直接通过注册的名称向进程发送消息，也可以通过 `whereis/1` 查询名称对应注册的进程ID
- `monitor/2` 可以监视一个进程，当监视的进程发生特点事件（比如退出）时，监视者会收到消息通知
- `link/1` 可以在父子进程之间建立双向连接关系，当一个进程异常退出时，另一个进程会收到退出信号，并因退出信号终止。可以通过 `process_flag(trap_exit, true)` 捕获退出信号，将退出信号转会为普通的信息进行处理

### 示例：冰箱

``` erlang
-module(kitchen).

-export([fridge/1, start/1, store/2, take/2]).

fridge(FoodList) ->
    receive
        {From, {store, Food}} ->
            From ! {self(), ok},
            fridge([Food|FoodList]);
        {From, {take, Food}} ->
            case lists:member(Food, FoodList) of
                true ->
                    From ! {self(), {ok, Food}},
                    fridge(lists:delete(Food, FoodList));
                false ->
                    From ! {self(), not_found},
                    fridge(FoodList)
            end;
        terminate ->
            ok
    end.


start(FoodList) ->
    % ?MODULE 是返回当前模块名的宏
    spawn(?MODULE, fridge, [FoodList]).


store(Pid, Food) ->
    Pid ! {self(), {store, Food}},
    receive
        {Pid, Msg} -> Msg
    after 3000 ->
        timeout
    end.

take(Pid, Food) ->
    Pid ! {self(), {take, Food}},
    receive
        {Pid, Msg} -> Msg
    after 3000 ->
        timeout
    end.
```

运行：

    % erl
    1> c(kitchen).
    {ok,kitchen}
    2> Pid = kitchen:start([]).
    <0.91.0>
    3> kitchen:take(Pid, hotdog).
    not_found
    4> kitchen:store(Pid, hotdog).
    ok
    5> kitchen:take(Pid, hotdog).
    {ok,hotdog}

### 示例：消息栈

``` erlang
-module(msg_cache).

-export([start_one/1]).
-export([loop/1]).

-record(state, {
            name,        %% 消息栈的名字
            length = 0,  %% 消息栈长度
            buff = []    %% 消息栈的存储列表
         }).

% loop 循环
% 1. 从 From 获取消息
% 2. 处理消息，发送响应给 From
% 3. 更新 State，进行下一个 loop 循环
loop(State = #state{name = Name, length = Len, buff = Buff}) ->
  receive
    {get_name, From}->
      From ! {ok, Name},
      loop(State);
    {get_length, From}->
      From ! {ok, Len},
      loop(State);
    {push, Msg, From} ->
      From ! ok,
      loop(State#state{name = Name, buff = [Msg | Buff], length = Len + 1});
    {pop, From} ->
      case Buff of
        [] ->
          From ! {error, empty},
          loop(State);
        [TopMsg | Msgs] ->
          From ! {ok, TopMsg},
          loop(State#state{name = Name, buff = Msgs, length = Len - 1})
      end;
    _Unsupported ->
      erlang:error(io_libs:format("unsupported msg: ", [_Unsupported]))
  end.

start_one(BuffName) ->
  %% 启动一个消息栈，并返回其 PID
  Pid = spawn(msg_cache, loop, [#state{name=BuffName, length=0, buff=[]}]),
  io:format("Buff ~s created! Pid = ~p~n", [BuffName, Pid]),
  Pid.
```

编译：

    erlc -o ebin msg_cache.erl

运行：

    3> flush().

    4> PID ! {pop, self()}.
    {pop,<0.84.0>}
    5> flush().
    Shell got {error,empty}
    ok

    6> PID ! {push, "msg1", self()}.
    {push,"msg1",<0.84.0>}
    7> PID ! {push, "msg2", self()}.
    {push,"msg2",<0.84.0>}
    8> PID ! {push, "msg3", self()}.
    {push,"msg3",<0.84.0>}
    9> flush().
    Shell got ok
    Shell got ok
    Shell got ok
    ok

    10> PID ! {get_length, self()}.
    {get_length,<0.84.0>}
    11> flush().
    Shell got {ok,3}

## 分布式 Erlang

每一个 Erlang VM 可以视为一个 Erlang Node，Erlang Node 之间可以互相连接，组成一个集群。

连接 2 个 Node：

- 通过调用 `net_kernel:connect_node/1` 函数。
- 在使用 RPC 调用一个远程方法时（比如 `rpc:call/4` 函数），自动加入集群。

### 演示

在本地启动 2 个 Node，设置相同的 cookie，分别命名为 node1, node2

    erl -name node1@127.0.0.1 -setcookie 'dist-cookie'

    erl -name node2@127.0.0.1 -setcookie 'dist-cookie'

在 node2 中连接 node1

    (node2@127.0.0.1)1> nodes().
    []
    (node2@127.0.0.1)2> net_kernel:connect_node('node1@127.0.0.1').
    true
    (node2@127.0.0.1)3> nodes().
    ['node1@127.0.0.1']

查看 node1 shell 的 PID

    (node1@127.0.0.1)1> self().
    <0.90.0>

在 node2 中查看 node1 shell PID 的表示

    (node2@127.0.0.1)4> ShellNode1 = rpc:call('node1@127.0.0.1', erlang, list_to_pid, ["<0.90.0>"]).
    <11870.90.0>

在 node2 通过 PID 向 node1 发送消息

    (node2@127.0.0.1)5> ShellNode1 ! "Hi, I am node2".
    "Hi, I am node2"

在 node1 中查看消息

    (node1@127.0.0.1)2> flush().
    Shell got "Hi, I am node2"
    ok

### 原理

启动 erlang 时，系统会确保一个 epmd(erlang port mapping daemon) 已经启动。

epmd 监听 4396 端口，并记录本地所有 erlang node 开放的分布式端口。

    % lsof -i -n -P | grep TCP | grep epmd
    epmd      14809 mapan    3u  IPv4 0xf3a3475d32b63118      0t0  TCP *:4369 (LISTEN)
    epmd      14809 mapan    4u  IPv6 0x20283c68a4dc373a      0t0  TCP *:4369 (LISTEN)

### CAP

数据多个副本分布在不同节点上

- 一致性（Consistency）：所有节点上的数据副本保持一致，每个节点的修改必须成功同步到其他节点
- 可用性（Availability）：节点能正常对外提供服务
- 分区容忍性（Partition Tolerance）：容忍网络分区的发生，

CAP 无法同时满足，最多只能满足 2 个特性：

- CP：网络分区发生时，拒绝对数据的修改请求，保证了一致性，但是牺牲了可用性
- AP：网络分区发生时，各分区节点正常处理请求，不同分区之间可能出现数据不一致，但是保证了可用性
- CA：无法容忍网络分区，这种情况几乎不可能

节点故障与网络分区：

当某些节点不可达之后，可能是这些节点出现硬件故障，也可能是与这些节点之间的网络出现问题

- 硬件故障：需要将故障节点上的服务转移到其他正常节点，故障节点从死亡恢复之后可以重新接管之前的任务
- 网络分区：如果发生网络分区，不能简单的在可达节点上恢复不可达节点的任务

##### kitty_server2（定义自己的 my_server）

kitty_server 展示了一类程序的行为模式


                                   +- state -+
                                   |         |
                                   v         |
    spawn --> init -- state -->  loop -------+ ---> exit
                                  ^^
                                  ||
                                 calls


1. 进程初始化 init 产生初始 state
2. 进程进入 loop，等待新消息
    1. 根据消息进行处理，产生新 state，进入下一个 loop
    2. 不进入下一个 loop，进程结束

将通用的行为抽象为 `my_server2`

``` erlang
-module(my_server2).

-export([start/2, start_link/2, call/2, cast/2, reply/2]).

% 创建进程，启动 Module 的 init 方法
start(Module, InitialState) ->
    spawn(fun() -> init(Module, InitialState) end).

% 创建进程，启动 Module 的 init 方法
start_link(Module, InitialState) ->
    spawn_link(fun() -> init(Module, InitialState) end).


% 同步发送：
% 1. 将 Msg 发送给 Pid 进程
% 2. 等待 Pid 进程回复，并作为返回值
call(Pid, Msg) ->
    Ref = erlang:monitor(process, Pid),
    Pid ! {sync, self(), Ref, Msg},
    receive
        {Ref, Reply} ->
            erlang:demonitor(Ref, [flush]),
            Reply;
        {'DOWN', Ref, process, Pid, Reason} ->
            erlang:error(Reason)
    after 5000 ->
        erlang:error(timeout)
    end.

% 异步发送：
% 1. 将 Msg 发送给 Pid 进程
% 2. 不等待回复，直接返回 ok
cast(Pid, Msg) ->
    Pid ! {async, Msg},
    ok.


% 向 Pid 发送 Reply
reply({Pid, Ref}, Reply) ->
    Pid ! {Ref, Reply}.

% 1. 执行 Module 的init 方法，获取初始状态
% 2. 进入 loop 循环
init(Module, InitialState) ->
    loop(Module, Module:init(InitialState)).

% 接收消息：
% 1. 如果异步 Msg，调用 Module 的 handle_cast 方法
% 2. 如果是同步 Msg，调用 Module 的 handle_call 方法
loop(Module, State) ->
    receive
        {async, Msg} ->
             loop(Module, Module:handle_cast(Msg, State));
        {sync, Pid, Ref, Msg} ->
             loop(Module, Module:handle_call(Msg, {Pid, Ref}, State))
    end.
```

#### supervisor

管理和监督其他进程：

- 当进程失败时，根据策略对其进行重启
- 通过顶层监督者关闭整个应用程序。当顶层监督者终止时，它会对每个子进程调用 `exit(ChildPid, shutdown)`

supervisor 只需要一个 `init/1` 回调函数，返回

``` erlang
{ok, {{RestartStrategy, MaxRestart, MaxTime}, [ChildSpecs]}}.
```

`RestartStrategy` 表示重启策略，可以为：
- `one_for_one`：如果监督者下有多个工作者，其中一个工作者失败，那么只重启失败的工作者
- `one_for_all`：如果监督者下有多个工作者，其中一个工作者失败，那么重启所有工作者
- `rest_for_one`：工作者之间有依赖关系，如果一个工作者失败，那么依赖它的进程也会重启
- `simple_one_for_one`

在 `MaxTime` 内重启超过 `MaxRestart` 次，监督者会放弃继续重启，将工作者关闭，然后自杀。

`ChildSpecs` 表示子进程规范


``` erlang
{ChildId, StartFunc, Restart, Shutdown, Type, Modules}.
```

- `ChildId`：监督者内部使用的一个内部名称
- `StartFunc`：一个描述如何启动子进程的元组，格式为 `{M, F, A}`。启动函数必须符合 OTP 标准，并且在执行时与调用者链接。
- `Restart`：告诉监督者在子进程死亡时如何反应，可以取 3 个值
    - `permanent`：永久进程应该始终重启
    - `temporary`：临时进程不进行重启
    - `transient`：瞬态进程应该一直运行，直到正常终止（退出原因是 `normal`）。如果非正常终止，进程会重启，如果正常终止，进程不会重启。
- `Shutdown`：通过监督者关闭程序时，设置终止的截止时间，超过截止时间还未结束的进程会被暴力终止 `exit(Pid, kill)`
- `Type`：表示子进程是工作进程还是监督者
- `Modules`：包含一个 module 名的列表

除了 `init` 返回的静态定义，supervisor 还提供了可以动态管理子进程的函数：

- `start_child(SupervisorNameOrPid, ChildSpec)`
- `terminate_child(SupervisorNameOrPid, ChildId)`
- `restart_child(SupervisorNameOrPid, ChildId)`
- `delete_child(SupervisorNameOrPid, ChildId)`
- `check_childspecs([ChildSpec])`
- `count_children(SupervisorNameOrPid)`
- `which_children(SupervisorNameOrPid)`

应用程序文件 `<yourapp>.app` 包含一堆 Erlang 项，以 VM 可理解的方式定义应用程序。

``` 
{application, ApplicationName, Properties}.
```

- `ApplicationName`：一个原子
- `Properties`：一个 `{Key, Value}` 元组列表
  - `{description, ""}`：描述
  - `{vsn, ""}`：版本号
  - `{modules, []}`：应用程序向系统引入的所有 module 的列表
  - `{registered, []}`：应用程序注册的所有名称的列表
  - `{env, [{Key, Val}]}`：一个键值对列表，可以作为应用程序的配置，可以通过 `application:get_env(Key)` 或者 `application:get_env(AppName, Key)` 在运行时获取配置的值
  - `{maxT, Milliseconds}`：应用程序可以运行的最长时间，默认 `infinity`
  - `{applications, []}`：应用程序依赖的其他应用程序列表，例如 `kernel`, `stdlib`
  - `{mod, {CallbackMod, Args}}`：应用程序的回调模块，OTP 在启动应用程序时调用 `CallbackMod:start(normal, Args)`，在停止应用程序时调用 `CallbackMod:stop(StartReturn)`。不包含此回调模块定义的应用程序被称为库应用程序。

`application` 只需要 2 个回调函数：

- `start/2`：即程序定义需要的 `CallbackMod:start(normal, Args)` 函数
- `stop/1`：即程序定义需要的 `CallbackMod:stop(StartReturn)` 函数

`application` 提供 2 个函数：

- `start(AppName)`：启动应用
- `stop(AppName)`：停止应用

## ETS(Erlang Term Storage)

一个高效的内存数据库，包含在虚拟机中。

ETS 表以元组作为元素进行存储，可以指定某个位置的值作为 key，表的类型包括：

- set：key - value 类型集合，key 值唯一
- ordered_set：key - value 类型集合，key 值唯一，并且数据按 key 值排序（表中第一个元素 key 值最小，最后一个元素 key 值最大）
- bag：允许多个 key 相同的数据存在
- duplicate_bag：允许多个 key, value 相同的数据存在

ETS 表有控制进程的概念，默认情况下，表的拥有者（即启动表的进程）可以写入，但所有进程都可以读取。可以将权限设置为公共（所有进程都可以读写）或者私有（只有拥有者可以读写）。

函数：

- `ets:new/2`
- `ets:delete(Table)`
- `ets:delete(Table, Key)`
- `ets:insert(Table, ObjectOrObjects)`
- `ets:lookup(Table, Key)`

DETS 是可以在磁盘（Disk）上持久化数据的 ETS 表
