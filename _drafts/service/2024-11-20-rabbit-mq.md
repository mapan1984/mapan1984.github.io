# RabbitMQ 基础使用

## 基本概念

* Publisher（生产者）：发送消息的应用程序或服务，将消息发布到 RabbitMQ 交换器
* Exchange（交换器）：生产者将消息发送到交换器后，由交换器决定将消息路由到哪些队列
* Queue（队列）：存储消息的地方
* Consumer（消费者）：接收并处理消息的应用程序或服务，消费者从队列中接收或拉取消息

消息流程概括如下：

```
                                  Server
                      +----------------------------+
                      |        Virtual host        |
                      |   +-------------------+    |
                      |   |     Exchange      |    |
    +-------------+   |   |     +-------+     |    |
    | Publisher   | ----------> |       |     |    |
    | application |   |   |     +---+---+     |    |
    +-------------+   |   |         |         |    |
                      |   |      Message      |    |
                      |   |       Queue       |    |
    +-------------+   |   |     +-------+     |    |
    | Consumer    | <---------- +-------+     |    |
    | application |   |   |     +-------+     |    |
    +-------------+   |   |     +-------+     |    |
                      |   +-------------------+    |
                      +----------------------------+

```

## 消息路由规则

Publisher 需要将消息发往 Exchange，由 Exchange 将消息路由到 Queue，整个过程概括如下：

1. Publisher 将消息发送到 Exchange，同时需要指定一个 **Routing Key**
2. Exchange 有与之绑定的 Queue 列表记录，RabbitMQ 中通过绑定将交换器与队列关联起来，在绑定时需要指定一个 **Binding Key**
3. Exchange 会将消息的 **Routing Key** 与 **Binding Key** 进行对比，如果匹配，就将消息路由到对应绑定的 Queue

根据 Exchange 类型的不同，Routing Key 与 Binding Key 的匹配规则也不同，Exchange 有以下类型：

- `Direct`（直连交换器）：交换器根据消息的 Routing Key 精确匹配队列。如果 Routing Key 与 Binding Key 一致，消息会被路由到对应的绑定队列。
- `Fanout`（扇出交换器）：交换器将消息广播给所有绑定的队列。消息的 Routing Key 在此类型中被忽略。
- `Topic`（主题交换器）：交换器根据消息的 Routing Key 模糊匹配队列。如果 RoutingKey 与 BindingKey 的模式匹配，消息会被路由到对应的队列。
    - 消息的 Routing Key 是由 `.` 符号分隔的单词列表
    - 绑定时指定的 Binding Key 也是由 `.` 符号分隔的单词列表，但是可以存在特殊字符 `*` 和 `#` 做模糊匹配
        - `*` 用于匹配一个单词
        - `#` 用于匹配零个到多个单词
- `Headers`（头交换器）：交换器根据消息的 header 属性进行路由（较少使用，且 **与上述规则不同** ）。
    - 消息的 headers 可以携带任意属性
    - 绑定队列时，指定 `x-match` 参数与其他需要匹配的 header 属性。
        - `x-match = all`：要求消息携带的 header 同时匹配所有 header 属性
        - `x-match = any`：要求消息携带的 header 任意符合一个 header 属性

## 开发

RabbitMQ 应用程序基本结构为：

1. 与 RabbitMQ 建立连接，创建 connection
2. 通过 connection 创建 channel
3. 通过 channel 声明 exchange
4. 通过 channel 声明 queue
5. 通过 channel 绑定（bind）exchange 和 queue
6. 通过 channel 执行 publish 向 exchange 推送消息
7. 通过 channel 执行 consume 从 queue 消费消息

### connection 连接

创建 connection，connection 表示客户端和服务端之间的 TCP 连接。

``` python
connection = pika.BlockingConnection(
    pika.ConnectionParameters(host="localhost"),
)
```

### channel 通道

channel 是基于 connection 创建的虚拟信道，同一个 connection 可以创建多个 channel，每个 channel 有自己的序号，通过序号区分不同 channel 的通信。

每个 channel 是独立的逻辑通道，操作互不干扰，因此一个应用程序可以只维护一个 connection，但通过多个 channel 进行不同任务处理（例如不同的业务、不同的线程），同时单个 channel 异常（例如 AMQP 协议错误）也不会影响其他 channel 或者整个连接。

channel 同时提供了对 exchange, queue, binding 等与 RabbitMQ 通信的 AMQP RPC 抽象方法，后续 exchange, queue, bind 的创建，以及消息 publish, consume 操作都需要通过 channel 方法执行。

``` python
channel = connection.channel()
```

### exchange 交换器

``` python
channel.exchange_declare(
    exchange="logs",
    exchange_type='direct',
    passive=False,
    durable=False,
    auto_delete=False,
    internal=False,
    arguments=None,
)
```

声明 exchange 参数：

- `exchange`：交换器名
- `type`：交换器类型
- `durable`：是否 **持久化**，即服务重启或继续存在
- `auto_delete`：是否在最后一个绑定的队列消失后删除交换器
- `internal`：设置后不能直接发布消息到此交换器，只能绑定到其他 exchange
- `arguments`：其他参数
  - `alternate-exchange`：设置 **备用交换器**，当交换器遇到无法路由的消息时，转发给备用交换器

RabbitMQ 提供了几个内置 exchange：

- `""`：空字符串 **默认 exchange**，类型为 direct，创建的每个队列都会自动以队列名称为路由键绑定到它。因此指定 exchange 为 `""`，routing key 为队列名，可以方便的直接将消息写入与 routing key 同名的队列。
- `amq.direct`：一个默认的 Direct Exchange。
- `amq.fanout`：一个默认的 Fanout Exchange。
- `amq.topic`：一个默认的 Topic Exchange。
- `amq.headers`：一个默认的 Headers Exchange。

### queue 队列

``` python
channel.queue_declare(
    queue='task_queue',
    passive=False,
    durable=False,
    exclusive=False,
    auto_delete=False,
    arguments=None,
)
```

声明 queue 参数：

- `queue`：队列名
- `durable`：是否 **持久化**，即服务重启后继续存在
- `exclusive`：是否 **独占**，仅由声明它的连接使用，当该连接关闭时，队列将被删除
- `auto_delete`：是否 **自动删除**，至少有一个消费者订阅的队列，在最后一个消费者取消订阅或消失时将被删除
- `arguments`：其他参数
  - `x-queue-type`：队列类型，默认 `classic`，可选 `quorum`, `stream`
  - `x-expires`：设置队列多少毫秒未使用后自动删除
  - `x-message-ttl`：设置消息投递进队列多久后被丢弃，队列中所有消息都有相同的 **TTL** (Time to Live)
  - `x-dead-letter-exchange`：设置 **死信交换器**
  - `x-max-length`：设置队列可以容纳的 **最大消息数**
  - `x-max-length-bytes`：设置队列可以容纳的 **最大字节长度**
  - `x-overflow`：达到队列容量限制后的溢出行为，默认为 `drop-head`（丢弃最久的消息），还可以设置 `reject-publish``reject-publish-dlx`（丢弃最近的消息，即刚发布的消息）
  - `x-max-priority`：设置队列支持的最高优先级

### bind 绑定

``` python
channel.queue_bind(
    exchange="logs",
    queue="info_logs",
    routing_key="info",
    arguments=None,
)
```

绑定 queue 与 exchange，参数：

- `exchange`
- `queue`
- `routing_key`
- `arguments`
  - `x-match`：指定 header 匹配类型，可选 `all`, `any`

``` python
channel.exchange_bind(
    destination='',
    source='',
    routing_key='',
    arguments=None,
)
```

除了通过 `queue_bind` 方法将队列绑定到交换器，还可以通过 `exchange_bind` 将一个交换器绑定到另一个交换器，允许创建更丰富的路由拓扑。

### publish 发布消息

``` python
channel.basic_publish(
    exchange="logs",
    routing_key="",
    body=message,
    properties=pika.BasicProperties(
        delivery_mode=pika.DeliveryMode.Persistent,
    ),
    mandatory=False
)
```

发布消息参数：

- `exchange`：交换器
- `routing_key`：路由键
- `body`：消息内容
- `mandatory`：要求至少将该消息路由到一个队列中，否则将消息返回给生产者
- `immediate`：如果该消息关联的队列上有消费者，则立即投递，否则将消息返回给生产者
- `properties`:
    - `content_type`：内容类型，由应用程序使用
    - `content_encoding`：内容编码，由应用程序使用
    - `headers`：可以携带 key - value 类型字典
    - `delivery_mode`：1(Transient) 或者 2(Persistent)，Persistent 表示 **持久化** 消息
    - `priority`：消息 **优先级**
    - `correlation_id`：关联ID，用于 **RPC 场景**，关联请求与响应
    - `reply_to`：回复响应队列名称，用于 **RPC 场景**
    - `expiration`：过期时间，用于设置消息的 **TTL** 每条消息的 TTL 可以不同
    - `message_id`：任意ID
    - `timestamp`：时间戳，由应用程序使用
    - `type`
    - `user_id`
    - `app_id`
    - `cluster_id`

> `immediate` 在 3.0 版本被删除 https://www.rabbitmq.com/blog/2012/11/19/breaking-things-with-rabbitmq-3-0#removal-of-immediate-flag

### consume/get 消费消息

``` python
def callback(ch, method, properties, body):
    print(f" [x] Received {body.decode()}")
    ch.basic_ack(delivery_tag=method.delivery_tag)

channel.basic_consume(
    queue='info_logs',
    on_message_callback=callback,
    auto_ack=False,
    exclusive=False,
    consumer_tag=None,
    arguments=None,
)
```

订阅队列 **（推模式）**，服务端通过连接主动推送消息，客户端注册消息处理回调函数

- `queue`：订阅队列
- `on_message_callback`：接收消息的回调函数，定义为 `on_message_callback(channel, method, properties, body)`
- `auto_ack`：是否自动确认消息
- `exclusive`：是否 **独占队列**，即不允许其他消费者订阅队列
- `arguments`
  - `x-priority`：消费者 **优先级**，优先向高优先级消费者推送消息，只有当高优先级消费者阻塞时，消息才会传递给低优先级消费者。未指定值的消费者优先级为 0。数字越大表示优先级越高，可以使用正数和负数。

``` python
channel.basic_get(
    queue,
    auto_ack=False
)
```

客户端主动从服务端拉消息，逐条获取 **（非常低效，不建议使用）**

- `queue`：订阅队列
- `auto_ack`：是否自动确认

#### ack/nack 消费确认

消费者可以通过 `auto_ack` 设置是否自动确认消息

- 自动确认模式下，消息确认是由 RabbitMQ 服务端自动完成的，即服务端在将消息分发给消费者后，会立即认为该消息已被成功处理并从队列中删除
- 手动确认模式下，消费者需要通过 `basic_ack` 手动确认消息已经处理

``` python
channel.basic_ack(delivery_tag=0, multiple=False)
```

确认消息：

- `delivery_tag`：标记消息的数字
- `multiple`：是否同时确认多条消息，如果同时确认多条消息，则 delivery_tag 之前所有消息都表示确认

``` python
channel.basic_reject(delivery_tag=0, requeue=True)
```

拒绝确认消息：

- `delivery_tag`：标记消息的数字
- `requeue`：是否 **重新进入队列**

``` python
channel.basic_nack(delivery_tag=0, multiple=False, requeue=True)
```

同 `basic_reject` 相比，可以同时拒绝多条消息

- `delivery_tag`：标记消息的数字
- `multiple`：是否同时拒绝多条消息，如果同时拒绝多条消息，则 delivery_tag 之前所有未确认消息都拒绝
- `requeue`：是否 **重新进入队列**

#### QoS（Quality of Service）

当多个消费者同时订阅同一个队列时，队列中消息将以轮询(round-robin)的分发方式发送给消费者，每条消息只会发送给一个消费者。在这种情况下：

1. 队列中只要有消息，就会发往消费者，可能导致消费者过载
2. 按次序轮询，不会考虑消费者是否空闲或者忙碌，可能导致消费者压力不均

在 ack 手动确认模式下，消费者可以通过 QoS 限制“正在传输”的消息数量（即在网络传输中或者已传递但未确认的消息），这可以避免消费者过载以及处理能力不均衡的问题。

``` python
channel.basic_qos(
    prefetch_size=0,
    prefetch_count=0,
    global_qos=False,
)
```

QoS 参数（Quality of Service）

- `prefetch_count`：在确认消息之前，消费者可以接收但尚未确认的最大消息数
- `prefetch_size`：在确认消息之前，消费者可以接收但尚未确认的最大大小，单位 Byte
- `global_qos`：false 表示是对消费者的限制；true 表示是对整个 channel 的限制

#### cancel 取消订阅

``` python
channel.basic_cancel(consumer_tag='')
```

取消订阅，取消订阅后，服务端不再向客户端发送消息：

- consumer_tag：标记消费者的字符串

## 高级特性

### 死信交换器/队列 DLX（Dead Letter Exchange）

消息在有些场景下可能会丢失，这种消息被称为死信：

- 消息被消费者通过 `basic_reject` 或者 `basic_nack` 拒绝，且 `requeue` 参数设置为 `false`
- 消息发布时设置了 `expiration` 属性，在队列中存在时间超过设置过期
- 队列设置了最大容量限制，消息因达到限制而被丢弃

当消息在队列中变成死信之后，为了避免丢失，可以给队列设置死信交换器（DLX），这样死信可以重新发布到 DLX 中，由 DLX 路由到其他队列中。

通过在 `channel.queue_declare` 方法中设置 `x-dead-letter-exchange` 参数来为队列添加 DLX

``` python
channel.queue_declare(queue="DLQ", durable=True)
channel.exchange_declare(exchange="DLX", durable=True, exchange_type="fanout")
channel.queue_bind("DLQ", "DLX")

channel.queue_declare(
    queue="hello_dlx",
    durable=True,
    arguments={
        'x-dead-letter-exchange': 'DLX',
    }
)
```

死信交换（DLX）是普通的交换器，可以是任何常见的类型，并像普通交换机一样声明。

### 延迟队列

延迟消息是指，消息发送后，并不希望消费者立即拿到消息，而是等待特定时间后，才能消费到消息。

延迟消息场景：

- 订单系统：例如用户下单后需要在 30 分钟内进行支付，如果 30 分钟内没有支付成功，对订单进行一次处理
- 远程控制设备在指定的时间后进行操作

RabbitMQ 本身并没有直接支持延迟队列，但是可以通过 DLX 和 TTL 实现延迟队列的功能。例如想实现延迟时间为 5s 的延迟队列：

1. 创建 TTL 为 5s 的队列 queue_5s
2. 为 queue_5s 绑定死信队列 queue_delay_5s
3. producer 向 queue_5s 发布消息
4. consumer 订阅 queue_delay_5s

``` python
# 1. 声明死信队列（queue_delay_5s）
channel.queue_declare(queue=DELAY_QUEUE, durable=True)

# 2. 声明 TTL 队列（queue_5s），并绑定死信队列
channel.queue_declare(
    queue=QUEUE_5S,
    durable=True,
    arguments={
        'x-message-ttl': 5000,  # 消息的 TTL 为 5 秒（5000 毫秒）
        'x-dead-letter-exchange': '',  # 使用默认交换机
        'x-dead-letter-routing-key': DELAY_QUEUE  # 死信队列的路由键
    }
)


# 3. 生产者向 queue_5s 发布消息
def producer():
    message = "Hello, this is a delayed message!"
    channel.basic_publish(
        exchange='',
        routing_key=QUEUE_5S,
        body=message,
        properties=pika.BasicProperties(
            delivery_mode=2  # 使消息持久化
        )
    )
    print(f"Producer: Sent message to {QUEUE_5S}: {message}")


# 4. 消费者订阅 queue_delay_5s
def consumer():
    def callback(ch, method, properties, body):
        print(f"Consumer: Received message from {DELAY_QUEUE}: {body.decode()}")
        ch.basic_ack(delivery_tag=method.delivery_tag)

    channel.basic_consume(queue=DELAY_QUEUE, on_message_callback=callback)

    print(f"Consumer: Waiting for messages from {DELAY_QUEUE}...")
    channel.start_consuming()
```

这样，当消息发布到 queue_5s 超过 5s 之后，变为死信进入 queue_delay_5s，才能被 consumer 消费到

### 优先级队列

支持通过设置队列的 `x-max-priority` 参数将队列设置为优先级队列，并且设定了本队列的最高优先级。

在发送消息时，通过设置消息的 `priority` 属性设置消息的优先级，消息的优先级默认最低为 0，最高为队列的优先级。在消息堆积在队列中的情况下，优先级高的消息具备优先被消息的特权。

支持 1 到 255 之间的优先级，但是，强烈建议使用 1 到 5 之间的优先级。较高的优先级值需要更多 CPU 和内存资源。

### RPC 实现（Remote Procedure Call）

RPC 流程为：

1. 客户端启动，创建一个匿名的回调队列，并订阅回调队列
2. 客户端为 RPC 请求设置 2 个属性：`replyTo` 表示服务端回复请求时的目的队列，即回调队列；`correlationId` 用来标识请求
3. 客户端将 RCP 请求发送到 `rpc_queue` 队列中
4. 服务端订阅 `rpc_queue` 队列，接收请求并处理，将处理结果返回给请求指定的 `replyTo` 队列，即回调队列
5. 客户端订阅回调队列，接收处理响应，根据响应的 `correlationId` 匹配请求

客户端示例：

``` python
import uuid

import pika

connection = pika.BlockingConnection(
    pika.ConnectionParameters(host="localhost"),
)

channel = connection.channel()

# 创建匿名回调队列
result = channel.queue_declare(queue="", exclusive=True)
callback_queue = result.method.queue

# 请求/响应关联ID
corr_id = str(uuid.uuid4())


# 订阅回调队列
def on_response(ch, method, props, body):
    if props.correlation_id == corr_id:
        print(method, props, body)


# 订阅回调队列
channel.basic_consume(
    queue=callback_queue,
    on_message_callback=on_response,
    auto_ack=True,
)

# 发送 RPC 请求
channel.basic_publish(
    exchange="",
    routing_key="rpc_queue",
    properties=pika.BasicProperties(
        reply_to=callback_queue,
        correlation_id=corr_id,
    ),
    body="hello",
)

# 等待响应
connection.process_data_events(time_limit=None)
```

服务端示例：

``` python
import pika

connection = pika.BlockingConnection(
    pika.ConnectionParameters(host="localhost"),
)
channel = connection.channel()

channel.queue_declare(queue="rpc_queue")


# 请求处理回调
def on_request(ch, method, props, body):
    response = f"ack {body}"

    # 收到请求后，将响应返回给 reply_to 队列，并设置响应的 correlation_id 为请求的 correlation_id
    ch.basic_publish(
        exchange="",
        routing_key=props.reply_to,
        properties=pika.BasicProperties(correlation_id=props.correlation_id),
        body=response,
    )
    ch.basic_ack(delivery_tag=method.delivery_tag)


# 订阅 rpc_queue 接收请求
channel.basic_qos(prefetch_count=1)
channel.basic_consume(queue="rpc_queue", on_message_callback=on_request)

channel.start_consuming()
```

```
              ------> rpc_queue  --->
      client                          server
              <------ reply_to  <----
```

### 生产者事务

- `channel.tx_select`：将当前 channel 设置为事务模式
- `channel.tx_commit`：提交事务
- `channel.tx_rollback`：回滚事务

``` python
channel.tx_select()

try:
    channel.basic_publish(exchange="", routing_key="hello", body="Hello World!")
except:
    channel.tx_rollback()
else:
    channel.tx_commit()
```

事务模式会显著降低 RabbitMQ 的吞吐量，如果需要更高性能，建议使用发布确认模式代替事务。

### 生产者确认

`channel.confirm_delivery()` 将当前 channel 设置为 publisher confirm 模式

``` python
channel.confirm_delivery()

try:
    channel.basic_publish(
        exchange='',
        routing_key='hello',
        body='Hello World!',
        properties=pika.BasicProperties(
            content_type='text/plain',
            delivery_mode=pika.DeliveryMode.Transient
        )
    )
    print('Message publish was confirmed')
except pika.exceptions.UnroutableError:
    print('Message could not be confirmed')
```

channel 设置为 confirm 模式后，在该 channel 发布的消息都会被指派一个唯一的 ID，当消息投递到匹配的队列后，服务端会发送一个包含 ID 的 ACK 给生产者
