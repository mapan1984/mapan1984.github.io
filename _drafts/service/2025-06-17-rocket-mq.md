# RocketMQ

## 概念

### 队列

队列的名称属性在 Apache RocketMQ 服务端的不同版本中有如下差异：
1. 服务端3.x/4.x版本：队列名称由{主题名称}+{BrokerID}+{QueueID}三元组组成，和物理节点绑定。
2. 服务端5.x版本：队列名称为一个集群分配的全局唯一的字符串组成，和物理节点解耦。

Apache RocketMQ 每个主题的多个队列可以分布在不同的服务节点上，在集群水平扩容增加节点后，为了保证集群流量的负载均衡，建议在新的服务节点上新增队列，或将旧的队列迁移到新的服务节点上。

### 消息

类型：
1. Normal：普通消息，消息本身无特殊语义，消息之间也没有任何关联。
2. FIFO：顺序消息，Apache RocketMQ 通过消息分组MessageGroup标记一组特定消息的先后顺序，可以保证消息的投递顺序严格按照消息发送时的顺序。
3. Delay：定时/延时消息，通过指定延时时间控制消息生产后不要立即投递，而是在延时间隔后才对消费者可见。
4. Transaction：事务消息，Apache RocketMQ 支持分布式事务消息，支持应用数据库更新和消息调用的事务一致性保障。

### 消费者

类型：
1. PushConsumer类型
2. SimpleConsumer类型
3. PullConsumer类型（仅推荐流处理场景使用）：使用PushConsumer类型的消费者消费消息时，消费者客户端必须设置消费监听器。Apache RocketMQ 服务端将消息推送给消费者后，消费者调用消息消费逻辑的监听器。

## 架构

- NameServer: 服务发现与负载均衡，允许客户端获取主题的数据分片和服务地址
- Proxy: 承载着消息的上层业务逻辑，尤其支持多种场景和协议
- Broker/Store: 负责核心消息存储，包括基于 Commitlog 的存储引擎、搜索索引、多副本技术以及与云存储的集成

Proxy 和 Broker 可以组合部署，也可以分开部署

### NameServer

Broker 启动后需要定时向 NameServer 发送心跳，将自身信息与主题分布上报到 NameServer。NameServer 需要记录心跳，并负责移除超时未上报的 Broker 信息。

Client 从 NameServer 首先获取主题的分布地址，然后通过地址连接对主题进行读写

RocketMQ中的路由消息是持久化在Broker中的，NameServer中的路由信息来自Broker的心跳包并存储在内存中。

### Storage

- commit log: 消息存储文件。broker 上所有主题消息会按序存储到同一个文件，每个文件大小默认 1GB，以物理 offset 作为文件名前缀。
- consumer queue: 消息消费的索引文件。存储路径为 `store/consumequeue/{topic}/{queue}/{filename}`，按主题/队列划分层级，当消息存储到 commit log 文件后，会将消息偏移量、大小等信息异步转发到 consumer queue 文件中，因此可以根据 topic、queue、偏移量等信息从 consumer queue 文件中定位消息的物理偏移量
- index files: 消息索引，存储消息 key 与 offset 的对应关系，RocketMQ 提供了按 key 查找消息的功能，

文件组织如下：

```
- ${ROCKET_HOME}/store/
  - commitlog/
  - conifg/
    - consumerFilter.json         # 主题消息过滤信息
    - consumerOffset.json         # 集群消费模式下的消息消费进度
    - delayOffset.json            # 延时消息队列拉取进度
    - subscriptionGroup.json      # 消息消费者的配置信息
    - topics.json                 # topic 配置信息
  - consumequeue/
  - index/
  - abort                         # abort 文件，指示 broker 是否非正常关闭
  - checkpoint                    # 存储 commitlog, consumerqueue, index 文件最后一次刷盘时间戳
```

### 高可用

#### Broker

##### Broker 组

Broker 组由 Master 与 Slave 组成，一个组下所有 broker 有相同的 `brokerName`，不同的 `brokerId`，其中 `brokerId=0` 代表 Master 节点，`brokerId>0` 代表 Slave 节点

- 当组中的 Master 失败时，Slave 仍然可以提供读服务，但是不可写（避免恢复后数据不一致吗？）
- 当一个组的 Master 和 Slave 都失败后，这一组 Broker 上所分布的 topic queue 不可读写，但是 topic 整体仍然可以读写，新消息会写入其他 queue。（与 kafka 的 topic partition 有差异）

##### DLedger Controller

DLedger Controller 可以与 NameServer 一起部署

4.5 开始支持 DLedger 模式，可以在主挂掉之后自动切主

#### NameServer

NameServer 是无共享架构，每个节点都是无状态的，因此，只要有一个名称服务器存活，整个服务发现机制就会正常运作，主题的读写可用性就不会受到影响。
