# 集群与高可用

RabbitMQ 可以通过搭建多节点集群，提高其性能和可用性。

> RabbitMQ 基于 Erlang/OTP 构建的，其集群组成、节点通信等功能，直接利用了 Erlang 自身的分布式能力，集群中每个 RabbitMQ 节点就是一个 Erlang （BEAM 虚拟机）节点。
> Erlang 原生支持分布式架构，运行在不同机器/设备上的 Erlang 虚拟机可以共同组成一个分布式系统，节点之间通过 TCP/IP 通信，在语言实现层面，内置了节点发现连接、远程调用等能力。

## 元数据

RabbitMQ 集群中所有节点都会备份所有的元数据信息，包括以下内容：

- 队列：队列的名称及属性
- 交换器：交换器的名称及属性
- 绑定关系：交换器与队列或者交换器与交换器之间的绑定关系
- vhost：为队列、交换器和绑定关系提供命名空间

元数据存储利用了 Erlang 自带的数据库 Mnesia，数据的多节点复制以及故障恢复也依赖于 Mnesia 自身的支持。
在创建交换器、队列、绑定关系等操作时，必须等待集群所有节点都成功变更元数据之后才返回。

> 4.0 版本之后，新增了 Khepri 后端存储支持，预计 4.1 版本之后 Khepri 会作为默认后端存储，4.2 版本之后将移除 Mnesia 支持。
> Khepri 使用 Raft 分布一致性协议，相比与 Mnesia 有更强的分区容忍性，可以避免发生网络分区后数据不一致以及手动恢复的情况。

### 交换器与绑定

交换器与绑定的本质是路由规则：

- 交换器其实只是一个名称和绑定列表。
- 当消息发布到交换器时，实际上是由所连接的信道将消息上的路由键同交换器的绑定列表进行比较，然后再路由消息。

交换器与绑定信息存在于所有节点，因此，连接任何一个节点，都可以访问到集群内所有的交换器。

### vhost

vhost 是逻辑概念，用于隔离资源与权限

- 同一个 RabbitMQ 集群中可以创建多个独立的 vhost
- 每个 vhost 有自己独立的队列、交换器、绑定和权限设置
- 在创建队列、交换器等资源时，必须指定在哪个 vhost 下操作
- 如果连接没有明确 vhost，使用 `rabbitmq.conf` 中配置的 `default_vhost = /`

> NOTE: vhost 更类似与命名空间 namespace 的概念，和节点毫无关系

## 数据分布

### 经典队列

队列与元数据不同，一个队列只存在一个节点上：

- 在 RabbitMQ 集群中创建队列，集群只会在单个节点而不是在所有节点上创建队列的进程，井包含完整的队列信息(元数据、状态、内容)。
- 这样只有队列的宿主节点，即所有者节点知道队列的所有信息，所有其他非所有者节点只知道队列的元数据和队列位于哪个节点。
- 因此当集群节点崩溃时，该节点的队列进程和关联的绑定都会消失。附加在那些队列上的消费者也会丢失其所订阅的信息，井且任何匹配该队列绑定信息的新消息也都会消失。

集群中队列声明与订阅：

- 默认情况下，当你在节点 A 上创建队列时，队列实际上是在节点 A 上创建的。
- 如果另一个客户端连接到节点 B，并声明相同的队列，节点 B 会检查集群中是否已经存在这个队列。
- 如果队列已经存在（在节点 A 上），节点 B 不会重新创建队列，而是会确认该队列的存在，并允许客户端继续操作。
- 另一个客户端如果在节点 B 上订阅该队列时，节点 B 会与队列所在的节点 A 进行通信，确保消息能正确路由到客户端。

为了使队列可以在集群节点间均匀分布，也可以使用参数 `x-queue-leader-locator` 决定声明队列时，队列的分布策略：

- `client-local`，默认值，将始终选择客户端连接到的节点
- `balanced`，它会考虑集群中每个节点上已运行的队列/领导者的数量；当集群中队列相对较少时，它会选择队列数量最少的节点；当队列很多（默认情况下超过 1000 个）时，它只会随机选择一个节点（计算确切数量在队列很多时可能会很慢，并且随机选择通常也一样好）

> 为了向后兼容，经典队列仍然支持 `x-queue-master-locator` 参数，允许不同的参数值：`client-local`、`random` 和 `min-masters`。后两个现在在内部映射到balanced。

### 镜像队列

RabbitMQ 集群只是提供多节点服务，共享元数据，但是存储数据的队列（经典队列）默认分布在单节点上。

在这种情况，集群多节点只能分散客户端连接压力，如果出现单节点故障，那么该节点上的队列也将不可用，无法做到高可用。

如果想实现队列的高可用，需要利用镜像队列功能，将队列镜像到集群的多个 Broker 之上，如果其中一个 Broker 失效了，队列能自动地切换到镜像中的另一个 Broker 上保证服务的可用性。

对于镜像队列：

客户端向镜像队列的主节点发送消息（通过 basic.publish），主节点负责将消息写入到自己的存储中。

主节点会将写入的消息同步到所有的镜像节点。同步机制是基于 RabbitMQ 的内部通信协议实现的，通常通过集群内部的网络传输完成。镜像节点会在本地存储消息的副本。



主节点会等待所有镜像节点确认消息已经同步成功。
如果所有镜像节点都成功存储消息，主节点会向客户端发送确认（ACK）。
如果任何一个镜像节点失败：
根据镜像队列的配置，可能会选择继续工作（异步模式），或者拒绝消息写入（同步模式）。
如果配置了 ha-sync-mode 为 automatic，镜像节点会在恢复后自动同步缺失的消息。


- 客户端向镜像队列的主节点发送消息，发送（Basic.Publish）到镜像队列的所有消息会被同时发往 master 和所有 slave 上，如果此时 master 挂掉，消息还会在 slave 上，这样 slave 提升为 master 的时候消息也不会丢失
- 除发送消息（Basic.Publish）外的所有动作都只会向 master 发送，然后在由 master 将命令执行的结果广播给各个 slave
- slave 会准确地按照 master 执行命令的顺序进行动作
- 如果 master 由于某种原因失效，那么加入时间最长的 slave 会被提升为新的 master
- 如果消费者与 slave 建立连接并进行订阅消息，那么 slave 会将消费请求转发给 master，master 准备好数据返回之后，再由 slave 投递给消费者。

镜像队列需要通过策略设置，策略的规则为：

    rabbitmqctl set_policy \
        --priority 0 \
        --apply-to queueus <policy_name> <pattern> <definition>

例如：设置策略对队列名称以 `queue_` 开头的所有队列进行镜像，并且在集群的 2 个节点上完成镜像

    rabbitmqctl set_policy \
        --priority 0 \
        --apply-to queueus \
        mirror_queue \
        "^queue_" \
        '{"ha-mode": "exactly", "ha-params": 2, "ha-sync-mode": "automatic"}'

`definition` 设置：

- ha-mode：镜像队列的模式
    - all：默认值，表示在集群中所有的节点上进行镜像
    - exactly：表示在指定个数的节点上进行镜像，节点个数由 ha-params 指定
    - nodes：表示在指定的节点上进行镜像，节点名称由 ha-params 指定
- ha-params：不同的 ha-mode 配置中需要用到的参数
- ha-sync-mode：队列中消息的同步方式
    - automatic：自动进行数据同步
    - manual：需要通过命令手动执行

> 镜像队列在 4.0 版本后已删除，应该使用仲裁队列或者流代替镜像队列

### 仲裁队列

仲裁队列是基于 Raft 共识算法实现的持久、复制的队列，应该被认为是在需要复制的高可用队列时的默认选择。

声明仲裁队列：

``` python
channel.queue_declare(
    queue="hello_quorum",
    durable=True,
    arguments={
        "x-queue-type": "quorum",
        "x-quorum-initial-group-size": 3,
    },
)
```

参数：

- `x-queue-type` 指定为 `quorum`
- `x-quorum-initial-group-size` 指定副本数，有实际意义的最小因子值是三个。建议因子为奇数，Raft 要求可用副本数至少为总数的一半以上。

每个仲裁队列都有一个 leader 和多个 follower，leader 节点故障时，副本节点会自动选举为新的 leader。

默认 leader 节点选择通过 `x-queue-leader-locator` 参数设置

- `x-queue-leader-locator`
    - `client-local`：声明是客户端连接的节点作为默认 leader 节点
    - `balanced`：如果总共少于 1000 个队列（经典队列、仲裁队列和流），选择托管最少数量的仲裁队列领导者的节点。如果总共超过 1000 个队列，则随机选择一个节点。

如果 leader 分布不均衡，可以重新平衡 leader 分布：

    rabbitmq-queues rebalance quorum

### 流

类似 kafka topic 的概念，数据持久保存，可以重复消费

声明流：

``` python
channel.queue_declare(
    queue="hello_stream",
    durable=True,
    arguments={
        "x-queue-type": "stream",
        "x-initial-cluster-size": 3,
    },
)
```

参数：

- `x-queue-type` 指定为 `stream`
- `x-initial-cluster-size` 指定副本数，有实际意义的最小因子值是三个。建议因子为奇数，Raft 要求可用副本数至少为总数的一半以上。

因为数据持久保存，为了避免磁盘写满，增加参数：

- `x-max-age`：保留时长
- `x-max-length-bytes`：保留大小
- `x-stream-max-segment-size-bytes`：段文件大小

默认 leader 节点选择通过 `x-queue-leader-locator` 参数设置

- `x-queue-leader-locator`
    - `client-local`：声明是客户端连接的节点作为默认 leader 节点
    - `balanced`：如果总共少于 1000 个队列（经典队列、仲裁队列和流），选择托管最少数量的仲裁队列领导者的节点。如果总共超过 1000 个队列，则随机选择一个节点。

消费时可以通过 `x-stream-offset` 指定消费位点

``` python
def callback(ch, method, properties, body):
    print(f" [x] Received {body.decode()}")
    ch.basic_ack(delivery_tag=method.delivery_tag)

channel.basic_qos(prefetch_count=1)
channel.basic_consume(
    queue="hello_stream",
    on_message_callback=callback,
    arguments={
        "x-stream-offset": "first",
    },
)
```

消费流时要求客户端必须连接到托管目标流副本（领导者或跟随者）的节点。

除了与队列相同的消费协议，RabbitMQ 增加了[专用的二进制协议](https://github.com/rabbitmq/rabbitmq-server/blob/main/deps/rabbitmq_stream/docs/PROTOCOL.adoc)来使用流中的数据，提供了对所有流特定功能的访问，并具有更高的性能。

启用流专用二进制协议

    rabbitmq-plugins enable rabbitmq_stream

## 集群组成

### 集群搭建

1. 配置节点 `/etc/hosts` 文件，让各节点可以通过节点名互相访问
2. 配置 `.erlang.cookie` 文件，确保集群各节点 cookie 一致
3. 启动第一个节点，让其余节点加入第一个节点，组成集群

### 节点类型

- disc：磁盘节点，元数据变更会写入磁盘，性能差，但重启不会丢失数据
- ram：内存节点，元数据变更只修改内存信息，性能高，但重启后会丢失数据。内存节点重启后，必须连接到预先配置的磁盘节点下载集群的元数据（内存节点唯一存储到磁盘的元数据信息时集群中磁盘节点的地址）。

RabbitMQ 要求集群中至少有一个磁盘节点，才可以执行队列、交换器、绑定关系等元数据的变更。

> 4.0 之后，内存节点类型支持将被移除

### 重启节点/集群

重新启动的节点将在启动时从其他对等节点同步模式和其他信息，在此过程完成之前，节点将无法完全启动并正常运行。如果对等节点不可用，则重新启动的节点将放弃并自愿停止。

当整个集群被关闭时，最后一个关闭的节点是唯一一个在关闭时没有运行中的对等节点的节点。该节点可以启动而无需尝试与任何已知对等节点同步，但是，它不会作为独立节点启动，并且对等节点将能够重新加入它。由于节点会尝试联系已知的对等节点长达 5 分钟（默认情况下），因此可以在这段时间内以任何顺序重新启动节点。这也要求重启集群时，第一个启动的节点为上一次最后关闭的节点。

## 网络分区

网络分区（Network Partition）是指在分布式系统，由于通信故障或其他原因，网络被分割成多个孤立的分区，每个分区中的节点只能与本分区内的其他节点通信，而无法与其他分区的节点通信。在这种情况下：

- 元数据：不同分区对交换器、队列、绑定的操作仅对当前分区有效，会导致不同分区节点之间的元数据不一致
- 镜像队列：每个分区中都会出现一个 master，不同分区的队列相互独立，如果同时有连接不同分区的客户端对队列进行读写，会导致不同分区队列数据不一致
- 经典队列：客户端连接一个分区的节点向 queue 发布/消费消息，如果 queue 实际位于另一个分区，当前分区的节点无法将消息转发过去，导致发布/消费失败

> 元数据后端 Khepri，以及仲裁队列、流等使用 Raft 共识协议，发生网络分区后，少于半数的节点会暂停服务，所以不存在网络分区不一致的问题。

### 判断

RabbitMQ 集群依赖 Erlang 原生的分布式能力，Erlang 节点之间使用周期性的心跳消息来监测节点状态。

如果节点之间网络故障，心跳交互中断超过 `net_ticktime` 会被认为节点掉线。

具体检查机制为：

* 节点之间定时发送心跳，每次心跳成功记一次 tick（默认 15 秒）
* 如果发生 4 次心跳失败，即超过 4 次没有记录 tick，则判断节点掉线（`net_ticktime` 必须是心跳延迟的 4 倍，因此默认 60 秒内没有收到心跳响应，节点会被标记为断开）

### 网络分区恢复

在网络连接恢复后，如果有分区数据不一致的情况，需要手动进行恢复：

1. 挑选信任分区，以这个分区的元数据作为集群元数据，丢弃其他非信任分区的改变
2. 重启非信任分区中的节点，重新加入信任分区
3. 重启信任分区中的所有节点以清除警告

选择信任分区的标准：

1. 分区中要有 disc 节点
2. 分区中的节点数最多
3. 分区中的队列数最多
4. 分区中客户端连接数最多

### 自动处理网络分区

RabbitMQ 提供了不同的分区处理策略，可以通过配置 `cluster_partition_handling` 进行设置：

- `ignore`：默认模式，不自动处理网络分区，分区之间互相隔离，独自运行
- `pause_minority`：当发生网络分区时，检测自身是否处于“少数派”（分区中的节点小于或者等于集群中一半的节点数），自动关闭这些节点的运行
- `pause_if_all_down`：如果当前节点和所配置的节点列表中的任何节点都不能交互，关闭当前节点
- `autoheal`：自动决定一个获胜的分区，然后重启不在这个分区中的节点来从网络分区中恢复
